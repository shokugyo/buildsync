# 1. 文書概要
## 1.1 文書名
BuildSync ソフトウェア外部設計書
## 1.2 対象システム
建設・建築業界向けクラウド型業務管理プラットフォーム「BuildSync」
## 1.3 文書目的
本書は、BuildSyncが提供する公開情報上の機能群をもとに、ユーザー視点で確認・合意すべき外部仕様を定義することを目的とする。
外部設計書として、主に以下を定義する。
- システムの利用目的
- 業務スコープ
- 利用者・権限
- 機能一覧
- 画面仕様
- 帳票仕様
- API仕様
- 外部インターフェース仕様
- 非機能要件
- セキュリティ要件
- 受入条件
## 1.4 外部設計書としての位置づけ
本書は、要件定義で定めた「何を実現するか」を、利用者が確認できる「画面・帳票・外部インターフェース・操作・入出力」の単位に具体化する文書である。
内部設計や詳細設計で扱う以下の事項は、本書の対象外とする。
- データベース物理設計
- クラス設計
- サーバー内部処理設計
- バッチ処理詳細
- インフラ構築手順
- ソースコード設計
- テーブルインデックス設計
- 実装フレームワーク詳細

# 2. 背景
## 2.1 建設業界における業務課題
建設・建築業界では、現場と事務所、元請会社と協力会社、営業と施工、経理と現場監督の間で情報が分断されやすい。
主な課題は以下である。
| 分類 | 課題 |
|---|---|
| 情報共有 | 電話、紙、FAX、メール、表計算ファイルが混在し、情報の所在が分からない |
| 工程管理 | 工程変更が協力会社・職人へ即時共有されない |
| 図面管理 | 最新図面と旧図面が混在し、誤施工リスクが発生する |
| 写真管理 | 工事写真の撮影、整理、台帳化に時間がかかる |
| チャット・連絡 | 現場ごとのやり取りが個人端末や個別アプリに分散する |
| 検査・品質 | 検査項目、指摘事項、是正状況の追跡が属人的になる |
| 受発注 | 発注書、請書、請求書、支払情報が案件と紐づかない |
| 原価管理 | 実行予算、発注額、請求額、支払額、粗利がリアルタイムに見えない |
| 営業管理 | 引合、商談、契約、施工、アフター情報が分断される |
| 人手不足 | 現場監督、営業、経理の事務作業負荷が高い |
| 法令対応 | 電子帳簿保存法、インボイス制度、個人情報保護への対応が求められる |
## 2.2 BuildSync導入の目的
BuildSyncは、建設業務に関わる情報をクラウド上で一元管理し、現場と経営の両面を可視化することを目的とする。
主な導入目的は以下である。
- 現場情報の一元化
- 施工管理業務の効率化
- 協力会社との情報共有強化
- 図面・写真・工程・チャットの統合管理
- 品質管理・検査管理の標準化
- 受発注・請求・原価管理の連携
- 経営判断に必要な案件別収支の可視化
- ペーパーレス化
- 業務履歴・証跡の蓄積

# 3. システム目的
## 3.1 システムの基本目的
BuildSyncは、建設プロジェクトに関する情報を案件・現場単位で管理し、関係者が同じ情報をリアルタイムに共有できるクラウドシステムとして機能する。
## 3.2 業務上の目的
| 目的 | 内容 |
|---|---|
| 現場管理の効率化 | 工程、図面、写真、検査、報告を一元管理する |
| 情報共有の迅速化 | チャット、通知、資料共有により関係者へ即時連絡する |
| 品質向上 | 検査項目、指摘事項、是正履歴を管理する |
| 原価可視化 | 発注、請求、支払、実行予算を案件単位で管理する |
| 経営判断支援 | 案件別売上、原価、粗利、進捗を可視化する |
| 協力会社連携 | 協力会社・職人がスマートフォンで必要情報を確認・報告できる |
| ペーパーレス化 | 図面、写真、帳票、証憑を電子化する |

# 10. 共通画面仕様
## 10.1 `SC-001` ログイン画面
### 10.1.1 目的
登録済みユーザーが認証を行い、権限に応じた機能へアクセスする。
### 10.1.2 表示項目
| 項目 | 種別 | 必須 | 説明 |
|---|---|---:|---|
| メールアドレス | テキストボックス | 必須 | ログインID |
| パスワード | パスワード入力 | 必須 | 認証用パスワード |
| ログイン状態を保持 | チェックボックス | 任意 | 次回ログイン補助 |
| ログイン | ボタン | - | 認証処理を実行 |
| パスワードを忘れた場合 | リンク | - | 再設定画面へ遷移 |
| 利用規約 | リンク | - | 利用規約を表示 |
| プライバシーポリシー | リンク | - | 個人情報取扱方針を表示 |
### 10.1.3 入力チェック
| チェック | 条件 | メッセージ例 |
|---|---|---|
| 必須 | メールアドレス未入力 | メールアドレスを入力してください |
| 必須 | パスワード未入力 | パスワードを入力してください |
| 形式 | メール形式不正 | メールアドレスの形式が正しくありません |
| 認証 | 認証失敗 | メールアドレスまたはパスワードが正しくありません |
| 状態 | 停止ユーザー | このアカウントは利用できません |
### 10.1.4 処理
1. ユーザーがメールアドレスとパスワードを入力する。
2. ログインボタン押下時に認証する。
3. 認証成功時、ホーム画面へ遷移する。
4. 認証失敗時、エラーメッセージを表示する。
5. 多要素認証が有効な場合、認証コード入力画面へ遷移する。

# 33. セキュリティ要件
## 33.1 認証
- ユーザーごとにログイン認証を行う。
- メールアドレスまたはユーザーIDとパスワードで認証する。
- パスワード再設定機能を提供する。
- 必要に応じて多要素認証に対応する。
## 33.2 認可
- ロール単位で機能権限を制御する。
- 案件単位で閲覧・編集権限を制御する。
- 会社単位でデータアクセス範囲を制御する。
- 協力会社は自社に関係する案件のみ閲覧できる。
- 原価、粗利、発注金額などの機密情報は限定公開とする。
## 33.3 データ保護
- 通信は暗号化する。
- 顧客情報、案件情報、個人情報、請求情報を保護する。
- ファイルアクセスにも権限チェックを適用する。
- 削除は原則として論理削除とし、監査証跡を保持する。
## 33.4 操作ログ
| 操作 | ログ内容 |
|---|---|
| ログイン | ユーザー、日時、端末、IP |
| 案件登録・更新 | 操作者、対象、変更内容、日時 |
| 図面登録・削除 | 対象ファイル、操作者、日時 |
| 写真登録・削除 | 対象写真、操作者、日時 |
| 発注承認 | 承認者、日時、結果 |
| 請求承認 | 承認者、日時、結果 |
| 権限変更 | 変更者、対象者、変更内容 |
| ファイルダウンロード | 対象ファイル、操作者、日時 |

## 1. 背景と目的
### 1.1 背景
建設・建築業界においては、現場の施工管理、協力会社とのコミュニケーション、受発注や請求などのバックオフィス業務において、依然として電話、FAX、紙ベースの図面や帳票への依存度が高く、非効率な業務プロセスが課題となっている。また、時間外労働の上限規制（2024年問題）に対応するため、生産性向上と業務のデジタル化が急務とされている。
### 1.2 目的
本システム（BuildSync）は、元請け企業、協力会社、職人が一元的に情報を共有・管理できるクラウドプラットフォームを提供する。現場のペーパーレス化、コミュニケーションの円滑化、および受発注から請求までのプロセスを電子化することで、建設プロジェクト全体の業務効率化、コスト削減、およびコンプライアンス強化（電子帳簿保存法・インボイス制度対応）を実現することを目的とする。
---
## 2. システムスコープ
### 2.1 対象業務範囲
本システムがカバーする主な業務範囲は以下の通りである。
*   **施工・工程管理業務**: 案件の登録、工程表の作成・進捗管理。
*   **図面・写真管理業務**: 現場写真の撮影・整理・台帳化、最新図面の共有と閲覧。
*   **現場コミュニケーション業務**: チャットを通じた連絡、日報の作成・承認。
*   **品質・安全管理業務**: 検査シートに基づく検査、指摘事項の管理・是正指示。
*   **受発注・経理業務**: 電子発注、請書の受領、見積書・請求書の管理、実行予算・粗利管理。
### 2.2 対象外業務
以下の業務は本システムの基本スコープ外とし、必要に応じて外部システムとの連携により補完する。
*   詳細な会計仕訳および決算業務（会計ソフトへデータ連携）
*   高度なCAD図面の作成・編集業務（作成済みデータの閲覧のみ対応）
*   給与計算および詳細な勤怠管理業務

## 4. 画面仕様
### 4.1 共通仕様
*   **レスポンシブ対応**: PC用Webアプリ、iOS/Android用ネイティブアプリのそれぞれに最適化されたUIを提供する。
*   **ヘッダー/フッター**: Web版にはグローバルナビゲーション（案件、チャット、図面、受発注、設定）と通知ベルアイコンを常時配置。スマホ版はボトムナビゲーションを採用。
*   **入力チェック**: クライアントサイドでのリアルタイムバリデーション（必須項目、文字数、フォーマット）を実装。
### 4.2 主要画面一覧（Webアプリケーション / PC）
#### 4.2.1 W-01: 総合ダッシュボード画面
*   **概要**: ログイン直後のポータル画面。対応すべきタスクを集約。
*   **表示項目**: 未読チャット一覧、本日稼働中の現場リスト、承認待ちドキュメント（日報、請求書）、遅延タスクアラート。
*   **アクション**: 各項目のクリックで詳細画面へ遷移。
#### 4.2.2 W-02: 案件詳細ワークスペース画面
*   **概要**: 特定案件のすべての情報をタブ切り替えで管理する画面。
*   **タブ構成**:
    *   `基本情報`: 物件情報、地図、アサインメンバー。
    *   `工程表`: ガントチャートの表示・編集エリア。ドラッグ＆ドロップ操作に対応。
    *   `資料・図面`: フォルダツリー構造でのファイルエクスプローラー形式。
    *   `写真`: サムネイル一覧表示、絞り込み（日付、フォルダ）、一括ダウンロード。
#### 4.2.3 W-03: 受発注・請求管理画面
*   **概要**: 取引先ごとの発注状況、請求状況を一覧管理する画面。
*   **表示項目**: 協力会社名、案件名、発注金額、ステータス（未発注/発注済/請負済/請求済/承認済）、インボイス登録番号。
### 4.3 主要画面一覧（モバイルアプリケーション / スマホ・タブレット）
#### 4.3.1 M-01: チャット画面
*   **概要**: 案件ごとのコミュニケーションを行う画面。
*   **表示項目**: メッセージタイムライン、既読ユーザー確認。
*   **アクション**: テキスト入力、カメラ起動、写真/図面添付、音声入力サポート。
#### 4.3.2 M-02: 写真撮影・電子小黒板画面
*   **概要**: 現場で写真を撮影し、アップロードする画面。
*   **機能**: カメラのプレビュー画面上に電子小黒板をオーバーレイ表示。小黒板のテキスト（工種、測点など）はタップで編集可能。撮影後、指定フォルダへ自動または手動でアップロード。
#### 4.3.3 M-03: 図面ビューア・ピン留め画面
*   **概要**: 図面（PDF等）を閲覧し、指摘事項や写真を紐づける画面。
*   **操作**: ピンチイン/アウトによる拡大縮小。任意の場所をロングタップして「ピン」を配置し、指摘テキストや現場写真を追加。

## 5. 帳票仕様
システムから出力（PDF・Excel）可能な主要帳票を定義する。
### 5.1 R-01: 工事写真台帳
*   **出力形式**: PDF, Excel (`.xlsx`)
*   **レイアウト**: 国土交通省準拠フォーマット。A4縦、1ページあたり写真3〜4枚。
*   **出力内容**: 写真画像、撮影日時、電子小黒板情報（工事名、工種、測点、略図等）、備考欄。
*   **出力条件**: 案件詳細の「写真」タブより、出力対象の写真をチェックして出力実行。
### 5.2 R-02: 工程表
*   **出力形式**: PDF, Excel (`.xlsx`)
*   **レイアウト**: ガントチャート形式。A3横またはA4横。
*   **出力内容**: 案件情報、工期カレンダー（日次・週次・月次指定可）、タスク名、担当業者名、進捗バー。自社用と施主用（業者名非表示）のフォーマット切り替え機能。
### 5.3 R-03: 発注書・請書・請求書
*   **出力形式**: PDF
*   **レイアウト**: 法定要件（電子帳簿保存法、インボイス制度（適格請求書等保存方式））を満たす標準フォーマット。
*   **出力内容**: 発行元情報、宛先情報、取引年月日、品目、単価、数量、税抜金額、消費税額（税率ごと）、適格請求書発行事業者登録番号、電子署名タイムスタンプ。

## 6. API連携仕様
外部システムとのデータ連携インターフェースを定義する。
### 6.1 A-01: 会計ソフト連携データ出力
*   **連携方式**: CSVエクスポート、または対象会計ソフト（勘定奉行、freee等）の提供するWeb APIへのPOSTリクエスト。
*   **連携タイミング**: 月次締め処理時、または任意のタイミングでの手動実行。
*   **データ内容**: 仕訳データ（借方勘定科目、貸方勘定科目、金額、取引先コード、税区分、案件コード等）。
### 6.2 A-02: BuildSync 公開API (RESTful)
*   **連携方式**: HTTPS RESTful API。JSONフォーマット。
*   **認証方式**: OAuth 2.0 に基づく Bearer トークン認証。
*   **主なエンドポイント**:
    *   `GET /api/v1/projects`: 案件一覧の取得
    *   `POST /api/v1/projects`: 新規案件の登録（SFAからの自動連携を想定）
    *   `GET /api/v1/users`: ユーザー一覧の取得
### 6.3 A-03: クラウドストレージ同期
*   **連携対象**: Box, Dropbox, Google Drive
*   **連携方式**: OAuth認証による対象サービスのAPI呼び出し。
*   **仕様**: BuildSync内の特定案件の「資料」フォルダと、外部ストレージの指定フォルダを定期的に双方向同期する。

## 7. 非機能要件
### 7.1 可用性・信頼性
*   **稼働時間**: 24時間365日。
*   **目標稼働率 (SLA)**: 99.9%以上（計画メンテナンス時間を除く）。
*   **冗長化**: クラウドインフラ（AWS等）を利用し、Webサーバー、アプリケーションサーバー、データベースサーバーをマルチAZ（アベイラビリティゾーン）構成とする。
### 7.2 性能・拡張性（パフォーマンス）
*   **レスポンスタイム**: Web画面の初期描画は原則3秒以内。写真アップロードや図面表示など、大容量データを扱う操作は非同期処理またはCDNを活用し、体感速度の低下を防ぐ。
*   **スケーラビリティ**: アクセス集中時（始業時、終業時の日報提出タイミング等）には、オートスケーリング機能により自動的にサーバリソースを拡張する。
### 7.3 セキュリティ
*   **通信の暗号化**: 全ての通信経路においてTLS 1.2以上（HTTPS）を利用する。
*   **データの暗号化**: データベース内に保存されるパスワード、個人情報、およびストレージ上のファイル群は暗号化して保存する（AES-256等）。
*   **認証・アクセス制御**: パスワードの複雑性要件、二段階認証（2FA）、IPアドレス制限機能を提供する。また、SAML 2.0対応によるSSO（シングルサインオン）をサポートする。
### 7.4 ユーザビリティ
*   **モバイル対応**: スマホアプリは、地下の現場や山間部など電波の届かない（オフライン）環境下でも、キャッシュされた図面の閲覧や写真撮影、日報の下書き保存ができること。通信回復時に自動でバックグラウンド同期を行うこと。
*   **操作性**: 現場作業員が手袋をしたままでも操作しやすいよう、ボタンやタップ領域はOSのガイドラインに準拠した十分なサイズを確保する。

### 1. 背景
建設・建築業界では、元請負人、協力会社、職人、施主間の情報共有が紙書類、メール、電話、個別チャットツールに依存しており、情報散逸、コミュニケーション遅延、原価超過、書類不備、現場の非効率が慢性化している。  
BuildSyncはこれらの課題を解決するため、プロジェクト単位の仮想作業空間「BuildSyncルーム」を中核に、工程管理、黒板機能付き写真管理、リアルタイムチャット、書類・検査・原価管理をクラウド上で完全に一元化するSaaSプラットフォームとして開発された。  
モバイルファースト設計、AIを活用した写真自動分類・類似検知、現場作業員でも直感的に操作可能なUIにより、「現場が主役」の生産性向上を実現する。本外部設計書は、公式サイト（https://BuildSync.jp/、https://BuildSync.co.jp）および公開資料、IPA「外部設計書作成ガイドライン」を参照し、ユーザー視点の画面・帳票・API・非機能仕様を正式に定義するものである。

### 4. 画面仕様
全画面共通仕様：ヘッダー（プロジェクト切替・全文検索・通知ベル・ユーザーアイコン）、権限制御（RBAC＋プロジェクト単位）、レスポンシブデザイン、カラーパレット（メイン#0066CC、遅延赤、完了緑）、トースト通知、モバイルボトムナビゲーション。
#### 4.1 プロジェクト一覧画面
- **目的**: ユーザーが管理下の全プロジェクトを一覧・検索・素早くアクセス  
- **レイアウト**: 上部フィルターバー＋カード/テーブル切替表示  
- **主な入力項目**: 検索キーワード（工事名・施主名）、フィルタ（ステータス、工事種別、担当者、期間）  
- **出力項目**: 工事名、施主、住所、進捗率バー、工期、責任者アバター、未読チャット数バッジ、遅延タスク数、契約金額  
- **操作**: 新規プロジェクト作成ボタン、カードクリックでプロジェクトルームへ遷移、ソート・フィルタ適用  
- **エラーメッセージ**: 「検索条件に該当するプロジェクトがありません」「権限が不足しています」  
- **遷移**: プロジェクトルーム概要画面、新規作成モーダル
#### 4.2 プロジェクト概要画面（BuildSyncルーム トップ）
- **目的**: プロジェクト全体状況を一目で把握し、クイックアクションを提供  
- **レイアウト**: ヘッダー（工事情報・メンバー一覧）、KPIサマリカード4分割（進捗・原価・写真・未読）、最新情報ウィジェット（工程・写真・チャット）  
- **出力項目**: 全体進捗率、予算消化率、総写真枚数、遅延タスク数、最新アップロード写真グリッド  
- **操作**: 大型クイックボタン（「写真撮影」「日報作成」「チャットを開く」「検査実施」）  
- **遷移**: 各機能タブ（工程・写真・チャット等）へ
#### 4.3 工程管理画面
- **目的**: 工程表の作成・更新・進捗管理・遅延監視  
- **レイアウト**: タブ切替（ガントチャート／リスト／カレンダー）、左側WBSツリー、右側タイムライン  
- **入力項目**（タスク追加・編集モーダル）: 作業名称、担当者（ユーザー/協力会社）、予定開始・終了日、前後工程依存、予定工数  
- **出力項目**: ガントバー（進捗塗りつぶし、クリティカルパス赤強調）、依存関係矢印、遅延アラート  
- **操作**: ドラッグ＆ドロップ日程変更、タスククリックで詳細モーダル、工程表PDF出力、Excel取込  
- **遷移**: タスク詳細モーダル（チャット・関連写真表示）
#### 4.4 写真撮影フロー（モバイル最優先）
- **目的**: 現場で即時黒板写真を撮影・登録・分類  
- **ステップ**:  
  1. カメラ起動（全画面フローティングボタン）  
  2. リアルタイム黒板オーバーレイ（工事名・日時・撮影者・GPS住所自動入力・編集可）  
  3. タグ選択（部位・作業種別・検査区分）  
  4. 撮影 → AI分類提案（完了・不具合・安全・材料等）  
  5. 確認・コメント追加 → アップロード  
- **出力項目**: 合成済み黒板画像、メタデータ（GPS座標、AIタグ、撮影者、日時）  
- **エラーメッセージ**: 「位置情報が取得できません」「オフラインのため後で同期します」
#### 4.5 写真アルバム画面
- **レイアウト**: グリッド表示（サムネイルに黒板一部オーバーレイ）、フィルタサイドバー、地図表示切替  
- **入力項目**: フィルタ（日付範囲、AIタグ、部位、投稿者）、画像類似検索  
- **操作**: 写真クリックで詳細モーダル、複数選択一括タグ付け・ダウンロード
#### 4.6 チャット画面
- **レイアウト**: Slack類似（チャンネル一覧・メッセージリスト・関連オブジェクトパネル）  
- **入力項目**: テキスト入力、写真/ファイル添付、メンション、関連オブジェクト（タスク・写真・検査）リンク  
- **出力項目**: メッセージ一覧（既読表示・リアクション・埋め込みカード）、全文検索結果  
- **操作**: 任意メッセージにタスクや写真を紐付け
#### 4.7 その他主要画面
- 日報作成画面（モバイル最適化テーブル形式、天候・作業員・資材・写真添付）  
- 検査管理画面（チェックリスト＋カンバン形式不具合台帳）  
- 原価管理画面（予算ツリー＋実績表＋グラフ）  
- 全社ダッシュボード画面
（各画面に権限制御・オフライン対応・プッシュ通知連動を適用）
### 5. 帳票仕様
全帳票共通項目：プロジェクト名・工事住所・出力日時・出力者名・会社ロゴ・ページ番号・フッター（BuildSyncロゴ）

### 1. 背景
建設・建築業界では、元請負人、協力会社、職人、施主間の情報共有が紙書類、メール、電話、個別チャットツールに依存しており、情報散逸、コミュニケーション遅延、原価超過、書類不備、現場の非効率が慢性化している。  
BuildSyncはこれらの課題を解決するため、プロジェクト単位の仮想作業空間「BuildSyncルーム」を中核に、工程管理、黒板機能付き写真管理、リアルタイムチャット、書類・検査・原価管理をクラウド上で完全に一元化するSaaSプラットフォームとして開発された。  
モバイルファースト設計、AIを活用した写真自動分類・類似検知、現場作業員でも直感的に操作可能なUIにより、「現場が主役」の生産性向上を実現する。本外部設計書は、公式サイト（https://BuildSync.jp/、https://BuildSync.co.jp）および公開資料、IPA「外部設計書作成ガイドライン」を参照し、ユーザー視点の画面・帳票・API・非機能仕様を正式に定義するものである。

#### 4.4 写真撮影フロー（モバイル最優先）

  2. リアルタイム黒板オーバーレイ（工事名・日時・撮影者・GPS住所自動入力・編集可）

  5. 確認・コメント追加 → アップロード  
- **出力項目**: 合成済み黒板画像、メタデータ（GPS座標、AIタグ、撮影者、日時）
